# Project Review

Date: 2026-06-12
Repository: `meufis-kondaki`
Branch reviewed: `main`

## Executive Summary

This is a small, well-contained Svelte 5/Vite PWA with a simple static data
pipeline. The core app builds, the unit suite passes, the Playwright smoke
suite passes, dependency audit reports no vulnerabilities, and an actual
browser offline smoke test passed after service-worker activation.

No critical production-stopping defect was found. The main risks are deployment
ambiguity, CI not enforcing the existing tests before deploy, and a few
user-visible/documented feature mismatches in follow mode and search.

## Verification Performed

- `npm test`: passed, 7 test files and 32 tests.
- `npm run build`: passed. Built 10 text JSON files plus search/index data.
- `npm run e2e`: passed, 8 Playwright tests.
- `npm audit --audit-level=low`: 0 vulnerabilities.
- Generated data consistency check: 10 texts, 5 categories, 1,377,302 bytes of
  generated data, no duplicate ids, no missing files, no bad TOC anchors, no
  unknown block types or roles.
- Browser offline smoke test: passed. After SW activation, offline reload,
  opening an unvisited text, and offline search all worked.
- Browser spot checks:
  - Follow-mode first block font size stayed `14px` before and after entering
    follow mode.
  - Search results rendered no highlight markup for the matched phrase.

## Findings

### P1 - Deployment Target Is Ambiguous

`README.md:7-8` says hosting is Cloudflare Workers with `wrangler.jsonc`.
`README.md:76-77` says every push to `main` publishes `dist/` through
`.github/workflows/deploy.yml`. The workflow, however, only deploys to
`gh-pages` through `peaceiris/actions-gh-pages` (`.github/workflows/deploy.yml:29-33`).
There is no `wrangler deploy` step.

Impact: a maintainer can push to `main` believing `https://gulani.ge` was
published while only the GitHub Pages fallback was updated, unless Cloudflare
has an external integration not represented in this repo.

Recommendation: choose and document the primary deployment path. If Cloudflare
Workers is primary, add a CI deploy step using Wrangler and secrets, or document
the external Cloudflare integration explicitly. If GitHub Pages is primary,
update the README and DNS/runbook accordingly.

### P1 - CI Deploys Without Running Tests

The project has useful test commands in `package.json:10-12`, but the deploy
workflow only runs `npm ci` and `npm run build` (`.github/workflows/deploy.yml:24-27`).

Impact: regressions covered by the existing Vitest or Playwright suites can
still be deployed to `gh-pages`.

Recommendation: at minimum run `npm test` before publishing. Prefer also
running `npm run e2e` in CI, with the Playwright Chromium browser installed or
cached. Keep deploy gated on those checks.

### P2 - Follow Mode Does Not Actually Increase Reader Text Size

`REDESIGN_NOTES.md:141-142` says follow mode forces wake lock and bumps font
one step for distance readability. The implementation only sets `follow = true`
and wake lock in `src/views/Reader.svelte:157`; the CSS tries to scale
`.reader` via `.view.follow .reader { font-size: 1.1em; }`
(`src/views/Reader.svelte:372`). Most reader text is explicitly sized in the
block component with `font-size: calc(... * var(--fs))`
(`src/components/Block.svelte:28`, `:36`, `:38`), so the parent font size does
not affect it.

Observed result: the first reader block computed to `14px` before and after
entering follow mode.

Recommendation: introduce a follow-aware scale variable used by block styles,
for example `--reader-scale`, or have follow mode temporarily apply a derived
font scale without mutating the user's saved `mk:font` preference.

### P2 - Search Match Highlighting Is Documented But Missing

`REDESIGN_NOTES.md:96-97` says search snippets show the match highlighted.
`SearchOverlay.svelte` returns a plain string from `snippet()` and renders it as
plain text (`src/components/SearchOverlay.svelte:25-30`, `:55`). Browser
inspection confirmed the result HTML contains no `mark`, `em`, `strong`, or
span markup around the match.

There is also a matching mismatch: `searchIndex()` matches normalized text
(`src/lib/search.js:3-18`), but `snippet()` locates the query with raw
`toLowerCase().indexOf(...)`. Queries that match only after punctuation or
spacing normalization can produce a poor snippet location.

Recommendation: either implement highlighted snippets with safe escaped markup
and normalized-match mapping, or update the redesign notes to remove the
highlight claim. Add a small unit test for punctuation-normalized snippets.

### P2 - Node Runtime Requirements Are Not Pinned

The dependency set currently requires a recent Node 20+ line. Installed package
metadata shows:

- `vite@8.0.16`: `^20.19.0 || >=22.12.0`
- `@sveltejs/vite-plugin-svelte@7.1.2`: `^20.19 || ^22.12 || >=24`
- `jsdom@29.1.1`: `^20.19.0 || ^22.13.0 || >=24.0.0`

The workflow uses `node-version: 20` (`.github/workflows/deploy.yml:20-22`),
but the repo has no `.nvmrc`, `.node-version`, or `package.json` `engines`
field, and the README does not state the minimum Node version.

Impact: local contributors on older Node 20 releases, Node 18, or stale system
Node installs may see install/build failures that are not obvious from docs.

Recommendation: add an engine declaration and a version file, for example
Node `20.20.2` or a compatible `22.x`, and document it in the README.

### P2 - No Static Check, Lint, Or Format Gate

`package.json:5-12` has build, test, and e2e scripts, but no `check`,
`svelte-check`, lint, or format command.

Impact: Svelte accessibility warnings, unused reactive code, accidental style
drift, and template mistakes can survive until manual review or runtime.

Recommendation: add a lightweight `npm run check` using `svelte-check`, and
consider ESLint/Prettier once the codebase starts seeing regular edits.

### P2 - Dialog Accessibility Is Minimal

The search overlay and bottom sheets use dialog-like UI, but they do not trap
focus, do not expose `aria-modal`, do not close on Escape, and do not restore
focus to the triggering control. Examples:

- `src/components/SearchOverlay.svelte:39-44`
- `src/components/RoleSheet.svelte:9-23`
- `src/components/TocSheet.svelte:21-36`

Impact: keyboard and screen-reader users can lose context or tab behind the
active overlay.

Recommendation: add a small shared dialog behavior for focus capture, Escape
close, background inertness where supported, and focus restoration.

### P3 - Build Emits A Font CSS Warning

`index.html:14` references `./fonts/fonts.css`. The file is copied from
`public/fonts/`, so the built `dist/fonts/fonts.css` exists and runtime works.
However, Vite emits:

`./fonts/fonts.css doesn't exist at build time, it will remain unchanged to be resolved at runtime`

Impact: this is not currently breaking the app, but it creates warning noise in
build and e2e output.

Recommendation: use a Vite-recognized public asset reference, such as
`%BASE_URL%fonts/fonts.css`, or move the font CSS into `src` and import it
through the normal Vite graph.

### P3 - Workbox Precache Contains Duplicate Icon/Manifest Entries

The generated `dist/sw.js` precaches icons and `manifest.webmanifest` from both
the Workbox glob (`vite.config.js:14-16`) and the PWA manifest icon integration
(`vite.config.js:21-36`).

Impact: minor service-worker manifest bloat and a little extra review noise.

Recommendation: remove the duplicate source, for example by excluding manifest
icons from one path or narrowing `globPatterns`.

### P3 - `.vite/` Is Untracked But Not Ignored

Current git status shows `?? .vite/`. `.gitignore:1-9` ignores common generated
folders but does not include `.vite/`.

Impact: local Vite cache state can accidentally appear in commits.

Recommendation: add `.vite/` to `.gitignore`.

### P3 - Some Documentation Is Now Archival But Not Marked As Such

`MOBILE_APP_PLAN.md:3-16` describes the project as a vanilla `app/` directory
with `build.js` and `data.js`, which is no longer the current architecture.
The superpowers spec also still mentions old paths like `app/data/index.json`
and `build.js`. `REDESIGN_NOTES.md:1-4` is framed as a `feature/redesign`
summary, while the current reviewed branch is `main`.

Impact: future contributors can follow stale instructions and make changes in
paths that no longer exist.

Recommendation: mark these as historical documents, move them under an
`docs/archive/` folder, or update them to reflect the current Svelte/Vite
layout.

### P3 - `gulani-logo.png` Appears Unused

`gulani-logo.png` is tracked and is about 932 KB. Repository search found no
references to it in source, docs, config, or build output.

Impact: small but unnecessary repository weight unless the file is intended as
the canonical source artwork.

Recommendation: either document it as the source image for icons/splash assets,
or remove it from the app repo.

### P3 - Data Loading Failure Handling Is Inconsistent

Home and Reader have explicit failure states. Category view calls
`loadIndex().then(...)` without a catch (`src/views/Category.svelte:7-12`), and
SearchOverlay has no visible error state if `search-index.json` fails to load
(`src/components/SearchOverlay.svelte:10`, `:46-59`).

Impact: under a partial deploy, corrupted cache, or bad network state before SW
activation, users may see a blank category page or empty search overlay.

Recommendation: add small failure states and retry buttons to Category and
SearchOverlay, matching the Home/Reader pattern.

### P3 - Parser Coverage Could Be More Golden-File Based

`build.cjs` is the most domain-specific part of the project and relies on
heuristics for bold paragraphs, speakers, rubrics, hybrid sections, and curated
landmarks. Unit tests cover representative parser behavior, but they do not
assert full generated-output invariants for the real markdown sources.

Impact: a text edit can accidentally change block classification or TOC
coverage without failing tests, as long as the build still succeeds.

Recommendation: add a generated-data regression test that runs the parser
against real source files and asserts stable text ids, block counts within
expected ranges, non-empty TOCs, and key landmarks for each service.

## Positive Findings

- Runtime architecture is simple and appropriate: static JSON, hash routing,
  no backend, and no runtime dependencies.
- XSS risk from text rendering is low: `fmt()` escapes HTML before applying
  controlled emphasis markup (`src/lib/fmt.js:1-3`). Role icons use controlled
  constants, not user data.
- PWA configuration keeps the old `sw.js` filename and deletes old
  `kondaki-*` caches on startup (`src/main.js:8-11`), which is the right shape
  for upgrading installed clients.
- Generated data is ignored and regenerated by `npm run build`, which keeps
  source markdown authoritative.
- Unit coverage is focused on the right pure modules: parser, router, search,
  roles, persisted store, formatting, and grouping.
- E2E coverage exercises the most important flows: home/category/reader,
  legacy URL redirect, search deep link, role highlighting, follow mode, theme
  persistence, scroll restore, and auto-scroll.
- Offline precache is healthy. The built SW includes app shell, fonts, icons,
  all data JSON files, and `search-index.json`; browser offline smoke passed.

## Recommended Priority Order

1. Decide and fix the deployment story, then gate deploys with tests.
2. Fix follow-mode font scaling and search snippet highlighting, or update docs
   if those are intentionally out of scope.
3. Add Node version pinning and a `check` script.
4. Clean up documentation drift and `.vite/` ignore state.
5. Add accessibility behavior for overlays.
6. Add real-source generated-data regression tests.
